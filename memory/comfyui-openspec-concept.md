# ComfyUI + OpenSpec 工程化方案

> P工的核心诉求：用 OpenSpec 把 ComfyUI 生成模型的流程标准化、可复用、可协作

## 痛点分析（ComfyUI 当前困境）

| 问题 | 具体表现 |
|------|---------|
| 工作流散乱 | JSON 文件命名随意，版本混乱 |
| 参数靠记忆 | 采样器、步数、CFG 每次重新调 |
| 无法团队协作 | 工作流是二进制/半结构化，无法 diff/review |
| 质量无标准 | 出图质量全凭主观判断 |
| 复用性差 | 换个项目/客户就要从头搭 |

## 解决思路：OpenSpec 分层规范

### 第一层：基础规范（Spec）
每个独立工作流一个 spec，定义"这个工作流是什么"：

```markdown
# 文生图工作流规范

## 输入
- 正面提示词：必填，英文，风格标签格式
- 负面提示词：选填，预设模板 + 自定义补充
- 参考图：选填，仅用于 img2img 场景

## 参数标准
| 参数 | 默认值 | 范围 | 说明 |
|------|--------|------|------|
| 采样器 | euler_ancestral | - | 速度快，适合概念设计 |
| 步数 | 20 | 15-30 | 低于15细节不足 |
| CFG | 7.0 | 5-12 | 过高导致过饱和 |
| 分辨率 | 1024×1024 | - | SDXL 标准 |
| 种子 | -1 (随机) | - | 复现时固定 |

## 输出规范
- 格式：PNG，保留 workflow metadata
- 命名：{项目}_{日期}_{序号}_{种子}.png
- 存储：按项目/日期两级目录

## 验收场景
- Given 提示词"一只猫坐在月球上"
- When 使用默认参数生成
- Then 输出 1024×1024 PNG，画面清晰，主体明确
```

### 第二层：变更管理（Change）
每次修改工作流，必须走 change 流程：

```
changes/
└── 2026-03-28-flux-migration/
    ├── proposal.md    ← 为什么迁移：Flux 画质更好
    ├── design.md      ← 方案：替换模型、调整参数、对比测试
    ├── tasks.md       ← 步骤：下载模型→替换节点→调参→验证
    └── specs/         ← 变更后的规范差异
```

### 第三层：项目组合（Workflow Suite）
多个 spec 组成完整流水线：

```
specs/
├── txt2img/          ← 文生图
├── img2img/          ← 图生图（精修）
├── img2model-3d/     ← 图生3D
├── upscale/          ← 高清放大
├── batch/            ← 批量生产
└── quality-check/    ← 质量检查
```

## 对模型师的价值

1. **可复用**：客户A的概念设计规范 → 直接套用到客户B
2. **可追溯**：每次参数调整都有 change 记录
3. **可协作**：新同事看 spec 就能上手，不用口口相传
4. **可度量**：通过 scenario 定义量化质量标准
5. **可演进**：模型升级（SD→SDXL→Flux）走 change 流程，不丢经验

## 落地路径

| 阶段 | 内容 | 时间 |
|------|------|------|
| 个人规范 | 先给自己的主力工作流写 spec | 第1周 |
| 变更记录 | 每次调整走 change 流程 | 第2-3周 |
| 流水线整合 | 多个工作流组成 suite | 第4周 |
| 团队推广 | 标准化后分享给同事 | 未来 |
